⚠️
Problema #1
Sem contexto persistente
A IA só vê as últimas 3 a 8 mensagens. Não há memória de projeto, decisões do usuário ou observações do orientador. Cada conversa nova esquece tudo.
⚠️
Problema #2
PDFs ignorados no chat
Anexos são gravados mas o chat principal nem lê. Sem chunking, sem embeddings, sem RAG. O Gemini responde sem o conhecimento que o aluno enviou.
⚠️
Problema #3
ABNT e plágio são fake
A "ação ABNT" só pede ao Gemini para "ajustar coisas que pareçam ABNT". O índice de plágio é hardcoded em 4.2%. Risco de credibilidade.
Solução: manter o Next.js como front + auth + Stripe + CRUDs simples. Criar um backend FastAPI dedicado ao cérebro do produto: memória persistente, RAG com pgvector, ABNT determinístico, originalidade in-house e jobs assíncronos. Migração incremental, sem big bang. Frontend continua funcional o tempo todo.

04 — Módulos FastAPI

10 domínios, responsabilidades claras

core/
Config (Pydantic Settings), DB async (SQLAlchemy 2.0), JWT validation com segredo compartilhado, dependencies, middlewares.
projects/
Memória persistente: research question, metodologia, decisões, observações do orientador, status por capítulo, referências selecionadas.
documents/
Upload e ingestão de PDFs. Job Celery faz extração via unstructured, chunking semântico (300-500 tokens, overlap 50).
embeddings/
Geração de vetores via text-embedding-004 (Gemini, 768 dim, gratuito). Cache por hash em Redis.
rag/
Busca semântica com pgvector. Endpoints /rag/search e /rag/ground (citation grounding). Filtra por tcc_id.
agents/
Orientador (analisa estado, gera observações), Redator (RAG-augmented), Revisor (coesão, gramática). Cada um com prompt e contexto próprios.
abnt/
Validador determinístico (margens, fonte, citações inline, NBR 6023) + Exporter DOCX com python-docx.
originality/
Pipeline híbrido. In-house: similaridade vs. capítulos próprios + PDFs do projeto. Hook drop-in para Copyleaks/Turnitin no futuro.
versions/
Snapshots de capítulos, listagem, restore. Diff visual entre versões com diff_match_patch.
jobs/
Endpoints que enfileiram tasks Celery. Modelo JobRun para o front consultar status via polling, sem WebSocket.
05 — Memória e Contexto

Como o "Orientador" funciona — a peça-chave

A cada chamada de IA (chat ou ação), o backend monta o contexto nesta ordem fixa. Esta é a diferença entre "um ChatGPT genérico" e "um orientador que conhece seu TCC".

  1. Identidade do projeto — título, área, instituição, norma, objetivo, metodologia. Sempre presente.
  2. Memória estruturada — decisões do usuário (ex: "excluímos o cap. 6"), observações do orientador IA (ex: "Cap. 3 sem citação primária"), status atual de cada capítulo.
  3. Capítulo em foco — conteúdo completo do capítulo que o usuário está editando agora.
  4. RAG — top-5 chunks dos PDFs de referência mais relevantes para a pergunta do usuário.
  5. Histórico compactado — resumo automático das mensagens com mais de 7 dias + últimas 5 mensagens cruas.
  6. Pergunta do usuário — o pedido em si.
Compactação automática: mensagens antigas são resumidas em background por job Celery (summarize_old_messages) e o resumo é gravado em Message.summary. Isso evita estourar limite de tokens sem perder contexto histórico.

07 — Originalidade

Pipeline híbrido in-house + hook para API paga

  1. Usuário escreve um parágrafo no editor.
  2. Ao salvar (ou sob demanda), backend roda OriginalityChecker.check(text).
  3. Pipeline interno: chunking sliding-window 100 tokens → embedding → busca de similaridade no pgvector contra (a) capítulos anteriores do próprio TCC, (b) PDFs de referência do usuário, (c) corpus opcional de frases acadêmicas comuns.
  4. Trechos com similaridade > 0.85 são marcados como "alto risco" e gravados em OriginalityReport.
  5. UI destaca trechos em amarelo/vermelho com tooltip "X% similar a [fonte]".
  6. Botão "sugerir paráfrase" chama IA para reescrever mantendo a citação.
Camada in-house — MVP
Custo zero, qualidade boa para o projeto
Detecta repetição interna e cópia direta de PDFs enviados. Suficiente para o aluno trabalhar com ética. Funciona offline, sem dependência externa.
Hook para API paga — futuro
Plano VIP com Copyleaks/Turnitin
Interface OriginalityProvider permite trocar o backend sem mudar o resto do código. Plano VIP futuro pode chamar API paga; PRO continua in-house.
Limites éticos explícitos: nunca remover citação para "esconder" similaridade. Sempre sugerir paráfrase + manter fonte. Disclaimer claro no produto: "esta verificação não substitui Turnitin/Copyleaks".

10 — Riscos

Onde a dívida técnica pode nascer

🔴 Dois ORMs (Prisma + SQLAlchemy) com fontes de verdade conflitantes
Mitigação: Prisma continua dono do schema. SQLAlchemy lê via reflection; Alembic só gerencia tabelas exclusivas do FastAPI. Decisão a travar antes da Sprint 3.
🔴 Compartilhamento de segredo JWT entre serviços
Mitigação: testes de contrato que validam tokens em ambos os lados desde a Sprint 1. Rotação de segredo documentada.
🟠 Custo de embeddings se base de PDFs explodir
Mitigação: cache por hash de conteúdo, deduplicação, coluna embedding_model_version para reembedding controlado.
🟠 Memória do projeto crescer sem limite
Mitigação: limite de 50 decisões/observações; rotação automática (mais antigas viram resumo). Política definida antes da Sprint 4 do Dev A.
🟠 Duplicação de lógica de plano (limits) em Next + FastAPI
Mitigação: centralizar em endpoint /me/plan que ambos consultam. Tabela plan_config como fonte única.
🟢 Manter /api/chat legado no Next "por compatibilidade"
Mitigação: deprecar com prazo claro (1 release); remover do código depois. Sem rota legada por mais de 2 sprints.
🟢 Celery sem dead-letter queue
Mitigação: configurar DLQ desde o início; alertas Sentry para falhas. Critério de aceitação da Sprint 3 do Dev A.
